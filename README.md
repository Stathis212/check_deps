# Stack Monitor

Automatically monitors releases, end-of-life dates, and CVE security advisories for your frontend stack. Runs 3x daily via GitHub Actions, emails you when something needs attention, and serves a live dashboard via GitHub Pages.

No hardcoded dates. No config files. No manual work.

---

## What it monitors

| Package | Releases | EOL dates | CVEs |
|---|---|---|---|
| Sitecore JSS | GitHub Releases API | Scraped from Sitecore KB1004260 | GitHub Advisory DB |
| Sitecore Content SDK | GitHub Releases API | Scraped from Sitecore KB1004260 | GitHub Advisory DB |
| Sitecore XM Cloud | GitHub Releases API | Scraped from Sitecore KB1004260 | GitHub Advisory DB |
| Next.js | GitHub Releases API | endoflife.date API | GitHub Advisory DB |

CVE audit covers: `next`, `@sitecore-jss/sitecore-jss`, `@sitecore-jss/sitecore-jss-nextjs`, `@sitecore-content-sdk/nextjs`, `react`, `react-dom`

---

## How it works

On every run the workflow does four things in sequence:

1. **Scrape** — Playwright launches a headless Chromium browser, renders the Sitecore lifecycle KB page, extracts EOL dates, and commits `.sitecore-lifecycle-cache.json` to the repo
2. **Check releases** — GitHub Releases API is queried for new versions and major bumps across all packages
3. **Check EOL** — Sitecore dates come from the scraped cache; Next.js dates are fetched live from the endoflife.date API
4. **Check CVEs** — GitHub Advisory Database is queried for security advisories; any CVE published in the last 30 days is always surfaced regardless of whether it was seen before

## Folder structure

```
your-repo/
├── index.html                               # Live dashboard (GitHub Pages)
├── .sitecore-lifecycle-cache.json           # Auto-generated by scraper — do not edit
├── .dep-monitor-state.json                  # Auto-generated state tracker — do not edit
├── scripts/
│   ├── scrape-sitecore-lifecycle.js         # Playwright scraper for Sitecore KB page
│   └── check-deps.js                        # Main monitoring script
└── .github/
    └── workflows/
        ├── dependency-monitor.yml           # Runs checks 3x daily
        └── deploy-pages.yml                 # Deploys dashboard to GitHub Pages
```

> Files prefixed with `.` are auto-generated on every run and committed back to the repo by the workflow. Never edit them manually.

## Setup — step by step

### Step 1 — Create a GitHub repository

Create a new repository on GitHub. It can be private. No template needed.

---

### Step 2 — Add the files

Copy the following files into your repo, preserving the exact folder structure above:

- `index.html` → repo root
- `scripts/scrape-sitecore-lifecycle.js` → `scripts/` folder
- `scripts/check-deps.js` → `scripts/` folder
- `.github/workflows/dependency-monitor.yml` → `.github/workflows/` folder
- `.github/workflows/deploy-pages.yml` → `.github/workflows/` folder

---

### Step 3 — Enable GitHub Pages

1. Go to your repo on GitHub
2. Click **Settings** → scroll to **Pages** in the left sidebar
3. Under **Build and deployment → Source**, select **GitHub Actions**
4. Click **Save**

This only needs to be done once. The dashboard will be deployed automatically on every push after this.

---

### Step 4 — Enable email notifications

1. Go to [github.com](https://github.com) → click your profile avatar → **Settings**
2. Click **Notifications** in the left sidebar
3. Scroll to **Actions**
4. Enable **Failed workflows only** for email

The monitoring script exits with a failure code whenever it finds alerts (new release, EOL warning, or CVE). This triggers GitHub's built-in email to your GitHub account email address. When everything is clear, the workflow exits cleanly and you receive no email.

---

### Step 5 — (Optional) Add Slack notifications

If you want alerts in Slack as well:

1. In your Slack workspace, go to **Apps → Incoming Webhooks → Add to Slack**
2. Choose a channel and copy the webhook URL
3. In your GitHub repo, go to **Settings → Secrets and variables → Actions → New repository secret**
4. Name: `SLACK_WEBHOOK_URL` — Value: the webhook URL you copied

---

### Step 6 — Commit and push

Push all files to your `main` branch. GitHub Actions will automatically:

- Register both workflows
- Deploy the dashboard to GitHub Pages
- Run the monitor on its next scheduled time

---

### Step 7 — Run manually for the first time

Before waiting for the schedule:

1. Go to your repo → **Actions** tab
2. Click **Dependency Monitor** in the left sidebar
3. Click **Run workflow** → **Run workflow** (green button)

This builds the initial state and scrapes the Sitecore lifecycle page for the first time. Check the run logs to confirm everything worked.

---

### Step 8 — Find your dashboard URL

After the first successful deploy:

- Go to **Settings → Pages** — the URL is shown in a blue box at the top
- Or go to **Actions → Deploy to GitHub Pages** → click the latest run → the URL appears at the bottom under the deployment summary

The URL format is always:
```
https://<your-github-username>.github.io/<your-repo-name>
```

Bookmark it. It is your permanent live dashboard.

## Alert schedule

|---|---|
| Morning | 07:00 |
| Noon | 12:00 |
| Afternoon | 16:00 |

To change the schedule, edit the `cron` entries in `.github/workflows/dependency-monitor.yml`.

## Alert types

| Alert | Trigger |
|---|---|
| `[NEW RELEASE]` | A new version was published since the last check |
| `[MAJOR VERSION BUMP]` | The latest release crossed a major version boundary |
| `[EOL IN X DAYS]` | A package is reaching end-of-life within the next 90 days |
| `[EOL PASSED]` | A package has passed its end-of-life date |
| `[CRITICAL / HIGH / MODERATE / LOW CVE]` | A security advisory was published within the last 30 days |

CVEs are always re-surfaced within the 30-day rolling window. Missing a day's email does not mean missing a CVE.

## Troubleshooting

**The workflow doesn't appear in the Actions tab**
The workflow file must be on your default branch (`main` or `master`). Push the file there and wait a minute for GitHub to register it. If there is a YAML syntax error, GitHub silently ignores the file — check for any red banner at the top of the Actions tab.

**"Not Found" error on Deploy to GitHub Pages**
GitHub Pages has not been enabled yet. Complete Step 3 above, then re-run the workflow.

**Node.js deprecation warning**
The workflow is configured for Node 24 with `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true`. If the warning reappears after a GitHub runner update, bump the `node-version` in `dependency-monitor.yml` to match the current LTS version.

**Sitecore lifecycle data is empty**
The Sitecore KB page is JavaScript-rendered. If Sitecore changes the page structure, the scraper may stop finding the table. Check the `Scrape Sitecore lifecycle page` step in the Actions log for details. The rest of the monitor (releases, CVEs, Next.js EOL) continues to work normally even if the scraper fails.

**GitHub rate limit errors on the dashboard**
The browser dashboard calls the GitHub API unauthenticated, which allows 60 requests per hour. If you hit the limit, wait an hour and refresh. The GitHub Actions script uses `GITHUB_TOKEN` which has a much higher limit.

## Adding a new package

To monitor an additional package:

1. Open `scripts/check-deps.js`
2. Add an entry to the `PACKAGES` array (for Sitecore products) or `EOL_API_PACKAGES` array (for packages on endoflife.date)
3. If you want CVE monitoring for it, add the npm package name to `NPM_PACKAGES_TO_AUDIT`
4. Open `index.html` and add the same entry to the matching array at the top of the `<script>` block
5. Commit and push
